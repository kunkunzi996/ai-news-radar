# 微信公众号退订后历史消息残留：修复方案

> 状态：第三轮独立审查后已重构，待 Task 0 开工确认，尚未施工。
> 项目：`E:\AI-news-reader\ai-news-radar-run`
> 任务类型：已有项目 Bug 修复（涉及桥接数据契约、存量身份迁移和归档清理，风险高）。

## 一、问题与目标

用户在 WeRSS 后台删除「财联社」后，最新本机采集和桥接 JSONL 已不再包含它，但 Radar 的
`archive.json` 与页面全量时间流仍保留 20 条财联社旧消息。

本次目标：

1. 真正从 WeRSS 删除公众号后，在下一次可信完整采集通过全部门控时清理该公众号历史。
2. 仍存在的公众号全部保留；即使它本轮没有文章，或只是被停用，也不能误删历史。
3. 采集失败、快照缺失、身份不完整、文件不匹配或通道未启用时，一条历史都不删。
4. 当前财联社 20 条残留通过稳定 `feed_id` 被精确识别和移除，其余 69 条微信历史全部保留。
5. 清理前可审计、清理后可按记录 ID 精确恢复，不能用整份旧归档覆盖新数据。

## 二、已确认根因

当前数据链路只发布文章快照，没有发布“数据库里当前还存在哪些公众号”的权威状态：

```text
WeRSS 数据库
  -> 只导出当前文章 JSONL
  -> wechat-bridge
  -> Radar 合并新文章
  -> 旧 archive 永久保留
```

`we_mp_rss_jsonl` 是容器型通道。`config/online-sources.json` 只配置整个微信桥接入口，没有逐个列出
猫笔刀、财联社等公众号，因此不能把现有 B 站/抖音的配置名单清理逻辑直接套到微信。

最底层问题不是“文章没更新”，而是归档缺少可验证的逐公众号稳定身份和退订事实。

## 三、第三轮独立审查结论与调整理由

### 3.1 保留前两轮审查中成立的护栏

以下意见本身正确，继续保留：

- `generated_at` 不能参与业务判变，否则每轮采集都会产生 bridge 空转提交。
- JSONL 不变但订阅状态变化时，仍必须发布新的订阅快照。
- 空名单确认计数必须封顶，不能每轮增长制造提交。
- 清理必须绑定本轮微信通道真实启用、读取成功和文件校验成功。
- 必须有 `off/audit/on` 三态开关，默认和异常值都按 `off`。
- 必须修订 `CLAUDE.md` 的清理禁区，登记窄范围例外，禁止其它通道类推。
- 验收条数不能写死为 60，应按各公众号实际导出条数计算。

### 3.2 不再把 `/rss/fresh` 当权威删除名单

旧方案要求从 `/rss/fresh?limit=100` 生成快照，但真实代码有两个不同口径：

- `/rss/fresh` 查询 `Feed` 全表，不过滤 `status`，并受接口 `limit=100` 限制。
- 本轮真实采集使用 `DB.get_all_mps()`，只取 `status == 1`，并排除
  `MP_WXS_FEATURED_ARTICLES`。

单纯给 `/rss/fresh` 增加“达到 100 就 `complete:false`”虽然能防误删，却会让订阅数达到 100 后
清理永久失效；而且它仍无法表达“记录还在但已停用”与“记录已被真正删除”的区别。

因此新方案改为：同步脚本从 sidecar 数据库一次读取完整 Feed 状态，快照同时记录：

- `known`：数据库里仍存在的非系统公众号，不论启用或停用。
- `active`：其中 `status == 1`、本轮实际参与采集的公众号。

默认保留策略是“只有 Feed 记录真正不存在才视为删除”。`status=0` 只代表暂停采集，历史保留。
这样严格对应用户本次“删除财联社”的操作，也避免把临时停用误当退订。若以后产品要改成“停用也清历史”，
必须另开任务明确业务语义，不能在本 Bug 中顺带扩大删除范围。

### 3.3 取消长期 aliases 状态机，改为一次性精确迁移

旧方案让无 ID 历史按名称/aliases 匹配，并设计 `aliases_reset` 与人工确认流程。该设计存在三个问题：

1. 两个不同 feed_id 使用相同名称或 aliases 时，自动回填会产生歧义。
2. `--acknowledge-alias-reset` 只有开关，没有定义如何安全补回丢失别名，施工路径不完整。
3. 名称迁移长期存在，会让未来一次改名重新进入“猜身份后删除”的危险路径。

真实数据给出了更简单的治本条件：当前微信归档 89 条全部缺 ID，但本机完整 bridge Git 历史已经包含
四个公众号的 `account -> feed_id` 映射：

```text
财联社           -> MP_WXS_3893127105 -> 20 条
卡尔的AI沃茨     -> MP_WXS_3871977637 -> 22 条
猫笔刀           -> MP_WXS_3198966508 -> 24 条
数字生命卡兹克   -> MP_WXS_3223096120 -> 23 条
```

因此先执行一次“只补 ID、不删除”的数据迁移，可让 89 条历史全部拥有稳定身份。此后运行期只按 ID
判断；任何仍缺 ID 的记录都触发熔断，不再靠名称自动删除。这样可以彻底取消 aliases、
`aliases_reset` 和别名基线继承。

### 3.4 补全三文件校验契约

旧 bridge 的 `manifest.json` 只记录文章 JSONL 哈希，并没有订阅快照文件名和哈希。旧方案虽然称
JSONL、快照、manifest 为“三件套”，却没有定义 manifest 如何绑定快照。

新方案将 manifest 升级为 schema 2，由 manifest 同时绑定文章文件和订阅快照；订阅快照再反向记录
文章 SHA256，形成可验证的完整链。旧 schema 1 仍可读取文章，但绝不允许触发清理。

### 3.5 回滚改为精确回插

旧方案建议恢复清理前整份 `archive.json`。若清理后已经采集了新文章，整文件覆盖会把新数据一起弄丢。

新方案要求在 source status 中记录本次候选记录 ID、feed_id 和候选摘要；误删时只把这些记录从清理前
提交精确回插到当前 archive，再重建派生文件。当前新增的数据和其它通道数据一律不覆盖。

### 3.6 坏 JSONL 行必须止于入口，不能污染 180 天归档

旧解析器会接受缺少 feed_id 的 JSONL 行并生成 RawItem。若实现只在清理阶段熔断，该坏行仍会被 merge
写入 archive；之后每一轮都因“archive 存在无 ID 微信记录”继续熔断，最长可能等到 180 天归档修剪后
才自行恢复。

因此严格校验不能只决定“本轮删不删”，还必须决定“这行能不能入库”：任何缺 ID、ID 不在 active
集合或结构非法的行，都不得生成 RawItem，不得进入 archive；同时把微信 status 标记为失败并记录拒绝
行数与原因。其它合法行可以按现有容错路径继续采集，但本轮清理必须关闭。这样一次坏输入只影响本轮，
修好 bridge 后下一轮即可恢复，不会把故障固化半年。

### 3.7 ID 迁移必须与定时 Actions 隔离

迁移会修改 `data/archive.json`，而 `update-news.yml` 每 30 分钟也会生成并提交 `data/**`。Git 的正常
非快进保护通常会拒绝其中一方，不会自动静默覆盖；但这仍会造成迁移 push 或 Actions push 失败，人工
错误 rebase 还可能把旧数据快照带回。

归档 SHA256 只能证明 dry-run 与 apply 使用同一份本地输入，不能锁住远端 master。因此迁移必须在短维护
窗口完成：暂停 update-news workflow、等待运行中任务结束、锁定远端 HEAD、在同一基准上完成
dry-run -> apply -> 重建 -> commit -> 普通 push，最后恢复 workflow 并跑一轮真实 Actions 验证 ID
仍在。远端 HEAD 变化或普通 push 被拒绝时，禁止 rebase/force，必须基于最新远端重新执行整个迁移流程。

## 四、第一性原理与业务边界

本方案基于四条不可破坏的事实：

1. 本轮没有文章，不等于公众号已经删除。
2. `status=0` 与数据库记录不存在不是一回事；默认只把后者视为删除。
3. 删除判断必须使用稳定 `feed_id`，名称只允许用于一次性、可审计的历史 ID 迁移。
4. 任何无法证明名单完整、身份完整和文件同源的情况，都只能延后清理，不能猜测后删除。

第一性原理三问：

- 最底层原因：采集范围、数据库存在状态、归档保留策略三者没有显式契约。
- 是否必须改数据库/schema/API：不需要改 WeRSS 数据库结构，也不需要新增对外 API；只需让现有同步脚本
  读取数据库状态并发布一个小型桥接契约。
- 是否有更小且同样治本的方案：有。一次性补齐现有历史 ID，运行期只做 ID 集合差，不维护长期别名状态机。

## 五、推荐方案

### 5.1 同步脚本输出本轮数据库权威输入

扩展 `deploy/local/we_mp_rss_sync_once.py`，新增参数：

```text
--subscriptions-out <临时 JSON 路径>
--snapshot-only
```

脚本在同一轮中：

1. 从 sidecar 数据库一次读取全部 Feed 记录。
2. 排除系统源 `MP_WXS_FEATURED_ARTICLES`。
3. 通过单一纯函数 `derive_active_feeds()` 生成 `known` 集合，并按
   `status == 1 and faker_id != MP_WXS_FEATURED_ARTICLES` 派生 `active` 集合。
4. 把同一份 `active` 对象列表交给现有 `sync_feeds()` 抓文章。
5. 全部 active 公众号同步成功后，原子写入临时权威输入；部分失败或进程异常时不生成可发布输入。

数据库若在本轮中途被修改，本轮继续使用开始时取得的同一快照，最坏只是把删除清理延后一轮，不会误删。
下一轮会读取新状态。

正式路径不再调用 `/rss/fresh`，也不读取 `WE_MP_RSS_FEEDS`。因此不存在环境变量子集污染或 100 条截断。
调试单号必须使用独立测试命令/夹具，不能改变正式桥接产物口径。

`derive_active_feeds()` 的结果必须通过集成测试锁定为与 sidecar 当前 `DB.get_all_mps()` 返回的 feed_id
集合完全相等。以后升级 sidecar 时，只要上游过滤条件变化，该测试就必须失败并要求同步审查，不能让
两套 active 口径静默分叉。

现有 PowerShell 的 `-SkipSync` 明确定义为诊断模式：

- 调用同步脚本的 `--snapshot-only` 只读取数据库状态，权威输入标记 `complete:false`、
  `reason=sync_skipped`，不能冒充成功采集。
- exporter 可以消费该输入生成临时诊断 JSONL/快照，便于检查 sidecar 库里的旧文章。
- 诊断产物只留在 `%TEMP%`/日志中，不得复制、暂存、提交或推送到 bridge。
- 状态写 `skip_sync_diagnostic_only` 并以 warning 结束；若连诊断输入都无法生成，则直接告警，不运行 exporter。

### 5.2 bridge 新增订阅快照

新增：

```text
output/wechat/jsonl/wechat_subscriptions_latest.json
```

建议结构：

```json
{
  "schema_version": 1,
  "generated_at": "2026-07-16T02:02:54Z",
  "complete": true,
  "authority_source": "sidecar_db_feed_table",
  "retention_policy": "feed_row_exists",
  "active_policy": "status_1_excluding_featured_v1",
  "source_jsonl_sha256": "...",
  "known_count": 3,
  "active_count": 3,
  "empty_confirmations": 0,
  "feeds": [
    {
      "feed_id": "MP_WXS_3198966508",
      "account": "猫笔刀",
      "status": 1,
      "active": true
    }
  ]
}
```

约束：

- `feeds` 是所有仍存在的非系统公众号；`active=true` 仅代表本轮实际采集。
- feed_id、account 必须非空，feed_id 必须唯一；status 只接受明确支持的 0/1，未知状态值使
  `complete:false`；`active=true` 必须与 `status == 1` 一致。
- `active_policy` 必须是云端明确支持的版本；未知策略只能读文章，不能清理。
- `known_count`、`active_count` 必须与数组实际内容一致。
- exporter 只能消费同步脚本生成的结构化临时输入，不能自行重新发现名单。
- 只有同步脚本成功、所有 active 公众号文章读取成功、JSONL 校验成功时，才发布 `complete:true`。
- JSONL 每一行必须可解析并包含 feed_id，且 feed_id 必须属于快照的 active 集合。
- 快照记录最终 JSONL SHA256，云端不匹配时只读文章或直接报告错误，绝不清理。

删除最后一个已知公众号时，以 `known_count == 0` 计算空名单确认：

- 第一份可信空快照：`empty_confirmations=1`，不清理。
- 连续第二份可信空快照：`empty_confirmations=2`，才允许清空微信历史。
- 计数使用 `min(2, previous + 1)`，达到 2 后保持不变。
- 只要 `known_count > 0`，计数立即归零；全部公众号只是停用时仍不算空名单。
- 上一版快照只能在 `git pull --ff-only` 后从 BridgeRoot 读取。缺失或损坏时按计数 0 重新开始，
  不继承临时目录或其它文件中的状态。

### 5.3 manifest schema 2 完整绑定两份数据

bridge 根目录 `manifest.json` 升级为：

```json
{
  "schema_version": 2,
  "generated_at": "2026-07-16T02:02:54Z",
  "article_file": "output/wechat/jsonl/wechat_contents_latest.jsonl",
  "article_sha256": "...",
  "subscription_file": "output/wechat/jsonl/wechat_subscriptions_latest.json",
  "subscription_sha256": "...",
  "output_rows": 60,
  "known_feed_count": 3,
  "active_feed_count": 3,
  "max_items": 20
}
```

云端验证顺序固定为：

```text
manifest schema 2
  -> manifest 指定的两个相对路径必须留在 bridge 根目录内
  -> 两个文件实际 SHA256 匹配 manifest
  -> snapshot.source_jsonl_sha256 匹配文章文件
  -> 行数、known_count、active_count 与实际解析结果匹配
```

路径必须做规范化和根目录约束，禁止 `..`、绝对路径或软链接逃逸。Actions 显式传入 bridge 根目录，
不再让读取器从 JSONL 子目录猜 manifest 位置。

旧 schema 1 或缺少订阅快照时保持向后兼容：文章照常读取，清理状态记为 `contract_not_cleanup_capable`。

云端读取必须把“校验”和“生成 RawItem”分开：先逐行扫描完整 JSONL，校验所有非空行的 JSON 结构、
必填字段、feed_id 与 active 集合，再执行 URL 去重和 `max_items` 截取。任一坏行都加入
`rejected_rows`，绝不能生成 RawItem；status 置 `ok=false` 并记录首批行号/原因。合法行可以返回供文章
采集，但由于 status 失败，本轮清理入口必须熔断。禁止只扫描到 max_items 就停止，否则文件尾部坏行会
逃过校验并让 manifest 的完整性判断失真。

### 5.4 三文件语义判变与原子发布

三类变化分别计算：

```text
bridgeChanged = jsonlChanged || snapshotChanged || manifestChanged
```

- JSONL 按最终文件 SHA256 比较。
- snapshot 比较时排除 `generated_at`，feeds 按 `feed_id` 排序；`source_jsonl_sha256` 保留在语义中。
- manifest 比较时排除 `generated_at`；底层文件哈希、数量或 schema 变化都属于业务变化。
- 三者语义都不变时保持 bridge 文件字节不动，暂存区为空，走 `completed_no_change`。
- JSONL 不变但删除了一个长期不发文公众号时，snapshot 和 manifest 仍必须提交。
- 只有 `generated_at` 变化时不得改写任何 bridge 文件，避免定时提交风暴。

发布必须采用多文件事务：

1. 所有候选文件先写到 `%TEMP%`，完成 JSON/schema/哈希/路径校验。
2. 计算最终实际字节与哈希后再生成 manifest。
3. 为 bridge 中三个明确文件创建本轮备份，再逐个原子替换。
4. 任一替换、校验或精确暂存失败，恢复这三个文件到本轮前状态，不留下半套脏文件。
5. 只暂存三个明确路径，在同一个 Git commit 中推送；远端只会看到完整 commit。

### 5.5 一次性补齐存量无 ID 历史

新增只负责迁移的脚本，默认 dry-run：

```text
scripts/migrate_we_mp_archive_feed_ids.py
```

职责：

1. 从本机完整 `wechat-bridge` Git 历史读取所有已发布 JSONL，不依赖云端 depth=1 clone。
2. 构建 `account -> set(feed_id)` 映射；一个名称对应多个 ID 时立即熔断。
3. 只处理 `site_id=we_mp_rss_jsonl` 且缺少 `we_mp_feed_id` 的 archive 记录。
4. dry-run 输出基准 Git commit、归档输入 SHA256、可补记录、未匹配记录、冲突记录和迁移后覆盖率。
5. `--apply` 必须同时携带 dry-run 输出的基准 commit 与归档 SHA256；HEAD 或输入任一变化时拒绝执行。
6. apply 只允许新增 `we_mp_feed_id` 字段，不得删除记录或修改标题、链接、时间、来源等字段。
7. 写入使用临时文件和原子替换，并生成可审计报告。

89 条是 2026-07-16 的当前基线，不是执行时写死的验收数字。off 代码上线到迁移执行之间，新文章会自动
携带 ID 入库，因此真正硬条件是：执行当时 archive 内全部无 ID 微信记录 100% 可补、0 未匹配、
0 冲突、迁移过程 0 删除；已有 ID 的新记录保持不变。

在 `scripts/radar/common.py` 先把 `we_mp_feed_id` 加入 `PUBLIC_RAW_META_FIELDS`，保证以后新文章和
重新出现的旧文章会持续携带 ID。

进入 `on` 前，微信归档 ID 覆盖率必须为 100%。如果完整 bridge 历史缺失、名称发生一对多冲突或
仍有未匹配记录，停止上线并人工核对；不得回退到运行期按名称删除。

### 5.6 运行期只按 feed_id 清理

清理候选仅允许满足全部条件的记录：

```text
record.site_id == "we_mp_rss_jsonl"
and record.we_mp_feed_id 非空
and record.we_mp_feed_id 不在 snapshot.feeds 的 known feed_id 集合
```

明确行为：

- feed_id 仍在 `feeds`、但 `active=false`：保留历史。
- feed_id 在 known 集合、本轮 0 篇文章：保留历史。
- feed_id 不在 known 集合：列为真正删除候选。
- archive 中出现无 ID 微信记录：整轮清理熔断，不自动猜名称。
- 新增或重新添加同一稳定 feed_id：继续保留并按现有首采规则回填。

清理发生在 `merge_archive_stage()` 合并本轮 `raw_items` 之前，只过滤内存中的 archive，再走现有流程
统一重建 `latest`、`stories`、`brief` 等文件，禁止逐个手改多份数据文件。

### 5.7 三态开关、通道绑定与审计记录

新增：

```text
WE_MP_RSS_SUBSCRIPTION_CLEANUP_MODE=off|audit|on
```

- `off`：默认和紧急停用值；不计算候选、不删除，文章采集照常。
- `audit`：验证完整契约并计算候选，但 archive 字节不得变化。
- `on`：所有门控通过后自动清理。启用 `on` 表示用户授权以后按可信 feed_id 自动处理真实删除，
  不是只批准当前财联社这一批。
- 非法值按 `off`。

进入 audit/on 必须同时满足：

1. `ctx.we_mp_rss_jsonl_enabled is True`。
2. 本轮确实调用微信 bridge 读取器。
3. 微信 status 存在且 `ok is True`。
4. manifest、JSONL、snapshot 都由本轮 bridge clone 成功读取并通过完整校验。

source status 至少记录：mode、bridge commit、manifest/snapshot schema、known/active 数、`rejected_rows`、
首批拒绝行号/原因、候选 feed_id、候选来源名、候选 item_id、有序候选摘要 SHA256、实际删除数和所有
跳过/熔断原因。

### 5.8 必须保留的熔断护栏

命中任意一条，整轮微信历史清理跳过：

1. mode 为 `off` 或非法；`audit` 永远不写 archive。
2. 微信通道未启用、未执行、bridge clone 失败或读取 status 不成功。
3. manifest 缺失、schema 不是 2、结构非法或包含越界路径。
4. JSONL/snapshot 文件缺失，或实际 SHA256 与 manifest 不一致。
5. snapshot 的 JSONL SHA256 与实际文章文件不一致。
6. snapshot `complete` 不是 true，或 authority/retention/active policy 不是支持值。
7. feed_id/account 缺失、feed_id 重复、status 不是支持值、active 不是 known 子集，或计数字段与实际数组不一致。
8. JSONL 任一非空行无法解析、缺少 feed_id、feed_id 不在 active 集合，或行数与 manifest 不一致。
9. archive 存在任何无 `we_mp_feed_id` 的微信记录，或身份迁移有名称一对多冲突。
10. archive 有微信历史且 known 集合非空，但一条 known feed_id 都匹配不到，判定 ID 口径异常。
11. known 集合为空且 `empty_confirmations < 2`。
12. 同步或文章导出存在任何公众号失败。

熔断必须写入 source status，不允许静默跳过。禁止使用“候选占比超过多少”作为熔断条件，因为用户可能
真的删除了历史量最大的公众号。
