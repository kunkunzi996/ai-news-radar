## 1. 本轮目标

云端（GitHub 星标自动同步）改了 A 信源、本机改了 B 信源时，用户在本地设置页点「同步到线上」
能**自动合并并推送成功**，不需要手动 `git pull`、不需要重启页面、不需要重录本机改动。

只有**双方改了同一个 `source.id` 的同一字段且结果不同**时才停下，并在页面上说清楚
「哪个信源 / 哪个字段 / 本机改成什么 / 云端改成什么」。

任何无法证明安全的情况继续拒绝同步，绝不强制覆盖云端配置。

## 2. 本轮不做什么

- 不改 `fresh_git_preflight()`（GitHub 星标 apply / unbind 走的那条路），它继续严格拒绝云端配置变化。
- 不改 Actions 触发方式、不加数据库、不放宽强制覆盖权限。
- 不引入 `git pull --rebase --autostash`（CLAUDE.md 明令禁区）。
- 不在同步路径里做任何历史清理（退订清历史仍只由「保存」路径按现有规则触发）。
- 不做「双方改了同一信源时替用户自动裁决取哪边」。
- 不做多分支 / 非 master 支持。

## 3. 项目调查结果

### 3.1 根因

`scripts/radar/server/online_sources.py:1432 _assert_remote_data_only_advance()` 要求：
本机 HEAD 到刚 fetch 到的云端 OID 之间，**每个 commit 只能碰 `data/`**，并且

### 3.2 为什么不能靠 `git rebase` 解决

`_commit_and_push_operation()` 现在是「先在 `pre_head` 上 `commit-tree`，再 `git rebase --merge fetched_oid`」。
若云端也改了同一个 JSON 文件，rebase 走的是**逐行三方文本合并**：
`updated_at` 是同一行、两边都改、值必不同 → **每次必冲突**。
所以文本 rebase 走不通，必须在 Python 层按 `source.id` 做**结构化合并**，再用确定性方式造出提交。

## 4. 技术方案

### 4.1 总体：双通道，老路一行不改

```
点「同步到线上」
  └─ sync_saved_online_source_config()
       ├─ 现有校验（recovery / ETag / 派生 OPML / index 干净）        ← 不变
       ├─ fetch 云端，判定推进合法性（放宽版断言）                    ← 改
       └─ 云端两个信源文件 blob 有没有变？
            ├─ 没变  →  完全走现有路径（老代码一行不动）
            └─ 变了  →  「合并同步」通道（§4.3）
```

放宽后的断言 `_assert_remote_advance(root, base, fetched, *, allow_online_config)`：

- 线性、单父；
- 每个 commit 的改动路径集合必须**要么全在 `data/`，要么全在那两个信源文件里**；
- 出现第三种路径 → `remote_non_data_changes` 拒绝；
- `base` 不是 `fetched` 祖先 → `remote_history_diverged` 拒绝；
- `allow_online_config=False` 时行为与今天逐字节等价。

**`allow_online_config` 不给默认值**（必填关键字参数）——漏改任何调用点都是 `TypeError`，
而不是静默沿用旧语义。五个台账驱动的调用点统一走集中策略：

```python
def _remote_advance_policy(manifest) -> bool:
    return manifest["operation_kind"] == "merge_sync"
```

`fresh_git_preflight()` 显式传 `False`；`_manual_sync_git_preflight()` 显式传 `True`
（它是分流判定点，此刻还没有任何副作用）。

### 4.2 合并算法（纯函数，零副作用）

`merge_online_source_configs(base, local, remote) -> (merged_config, conflicts)`

输入是三份**已经过 `validate_online_config_schema` 的配置对象**，合并单位是 `source.id`。
生成的 `online_opmlrss` 容器不参与合并，最后由 `build_online_config()` 重新派生。

#### 顶层字段：不合并，只守门

合并开始前先断言 `base / local / remote` 三份的 `version` 与 `mode`
**全部等于当前规范值**（`"1.0"` / `"online-public-source-config"`）。
任何一份不等 → 直接 `online_sources_merge_conflict`，`kind="top_level"`，details 带出三边的值。

> 为什么这么严：`build_online_config()` `:661-666` 固定输出这两个值。
> 若允许它们参与合并，云端的顶层变更会被无声抹掉。将来真要升 version，走显式迁移，
> 不由合并偷偷完成。

`github_star_sync` 按下表的同一套规则判定，并叠加下文的托管投影硬断言。

#### 信源逐条规则

| base | 本机 | 云端 | 结果 |
|---|---|---|---|
| 无 | 无 | 有 | 采用云端（云端新增） |
| 无 | 有 | 无 | 采用本机（本机新增） |
| 无 | 有 | 有 | 相同 → 采用；不同 → **冲突** `added_both` |
| 有 | =base | 变 | 采用云端 |
| 有 | 变 | =base | 采用本机 |
| 有 | 变 | 变 | 逐字段：只有一边改 → 取那边；两边改同一字段且值不同 → **冲突** `field_diff` |
| 有 | 删 | =base | 删除 |
| 有 | =base | 删 | 删除 |
| 有 | 删 | 删 | 删除 |
| 有 | 删 | 变 | **冲突** `delete_vs_modify` |
| 有 | 变 | 删 | **冲突** `delete_vs_modify` |

#### 合并后三条硬校验（任一不过 → 拒绝，不写盘、不提交、不推送）

1. `validate_online_config_schema(merged, existing=True)` 通过；
2. `protected_online_config_projection(merged) == protected_online_config_projection(remote)`
   —— 合并结果的「GitHub 托管源 + binding」必须与云端**逐字节一致**。
   这一条同时兜住实现边界 4（星标同步不得借合并触发历史删除）与「本机旧配置覆盖云端托管状态」；
3. 合并结果重新渲染出的 JSON / OPML 成对一致。

#### 冲突项结构

```
{kind, source_id, source_name, field, local_value, remote_value}
```

`kind ∈ {top_level, binding, added_both, field_diff, delete_vs_modify}`。
`*_value` 只在字段是 `bool` 或 ≤80 字符字符串时带出，并先过 `check_public_text_safe()`；
否则置空只报字段名。最多 20 条，超出折叠为「等 N 项」。

### 4.3 合并同步的 git 编排（`operation_kind = "merge_sync"`）

**核心决策：先把合并结果造成 git 对象并推上去，本机仓库最后才回填。**

理由：云端是唯一真源、本机仓库是副本。先推后拉可以彻底消灭
「本机已提交但推送被拒 → 与云端分叉 → 之后每次同步都 `remote_history_diverged` 卡死」
这个不可自愈的死局。推送被拒时本机**一个字节都没动**，用户再点一次即可。

#### 台账（schema v2）

`files[p].before_sha256` = **本机候选 L** 的哈希（保持既有语义：操作开始前工作区里的内容）
`files[p].after_sha256` = **合并结果 M** 的哈希
新增只属于 merge_sync 的证明子对象：

```json
"merge": {
  "fetched_oid":   "<40-64 hex，合并所基于的云端 OID>",
  "base_oid":      "<40-64 hex，必须 == pre_head>",
  "remote_sha256": { "config/online-sources.json": "<64hex>",
                     "feeds/online-sources.opml":  "<64hex>" }
}
```

约束：`operation_kind != "merge_sync"` 时 `merge` 必须为 `null`；是 merge_sync 时三个字段必须齐备。
`ONLINE_OPERATION_SCHEMA_VERSION` 升到 **2**，且 `_validate_operation_manifest()`
**同时接受 v1 与 v2**（不做这条会在升级瞬间把正在进行的事务打死）；新建台账一律写 v2；
旧服务读到 v2 → `schema_version != 1` → 409 fail-safe。

三份内容的记号，全文统一：

| 记号 | 含义 |
|---|---|
| **B** | 基线 = `pre_head` 上的两个信源文件 |
| **L** | 本机候选 = 用户保存后、工作区里的内容 |
| **R** | 云端 = `fetched_oid` 上的内容 |
| **M** | 合并结果 |
| **C** | 合并提交，父节点固定为 `fetched_oid` |

#### 步骤

```
1  只读算合并 → 有冲突：抛 409，全程零副作用，结束

2  写台账（kind=merge_sync, schema_version=2, files.before=L, files.after=M, merge={...}）

3  用临时 index（GIT_INDEX_FILE=<gitdir>/ai-news-radar-merge-index）：
     read-tree fetched_oid → hash-object -w 两个合并 blob → update-index --cacheinfo → write-tree = T
     全程不碰真 index、不碰工作区
   校验：diff-tree(fetched_oid, T) 的路径集合 ⊆ 那两个文件；
         blob 哈希与 merge.remote_sha256 / files.after 双向核对
   （不复用 _manifest_changed_paths()，避免语义借用）
   try/finally 删除临时 index 文件

4  C = commit-tree T -p fetched_oid -m "配置：同步线上信源" -m <operation trailer>
   用 _verify_merge_operation_commit()（不复用 _verify_operation_commit）：
     C 单父且父 == merge.fetched_oid
     diff-tree(fetched_oid, C) 路径集合 ⊆ 那两个文件
     两个 blob 哈希 == files.after
     commit message 含且仅含一次 trailer
   台账写入 operation_commit_oid=C, stable_patch_id, phase=committed

4.5 未跟踪碰撞预检（见下「载荷说明」）；命中 → 零副作用拒绝

5  push C:refs/heads/master
     被拒（云端又动了）→ 删台账、报可重试错误、本机零改动、结束
     merge_sync 的「重试」语义 = 重新点同步（重算合并），不复用 _retry_owned_operation_push()

6  本机回填（见下）

7  删台账，返回 outcome=pushed + merged_summary
```

#### 步骤 6：本机回填（信源文件 L→M 一步到位，**不经过 B**）

> 为什么不用 `git merge --ff-only`：它拒绝在「本地已修改且两端不同」的文件上跑，
> 迫使先把工作区退回 B。而 B 中间态会让用户看到「我的改动没了」，
> 是本计划明确列为缺陷的状态。改用按路径检出即可根除。

```
6a 门禁（主流程与恢复动作共用同一段代码，禁止复制两份）：

    ① 工作区两个信源文件哈希 ∈ {L, M}          ← 两态、单调、幂等
    ② HEAD ∈ {pre_head, C}
    ③ 台账 digest 与读入时一致
    ④ stash 校验 —— 必须按阶段选谓词，不能一把梭：
         stash 尚未建立（stash.oid 为空且 paths 为空）
             → 不做 stash 校验；6b 建 stash 前才用 _stash_path_matches() 核对
         stash 已建立（stash.oid 非空）
             → _owned_stash_oid() 证明归属 + _stash_restore_can_continue() 判定能否继续
               （该组合器已同时覆盖「还原目标干净」与「已还原」两种情形）
         stash 已还原并 drop（oid 与 paths 均被清空）
             → 不做 stash 校验

    任一不过 → 不写盘、不移 ref、不删台账，pending 且 allowed_actions=[]

6b stash 脏的非信源路径（若尚未 stash）。沿用现有 proof 机制与 --keep-index

6c changed = git diff --name-status -z --no-renames <pre_head> <C>
     只允许 A / M / D / T 四种状态，出现其它字母 → 停
     断言路径集合 ⊆ (data/** ∪ 那两个信源文件)，否则 → 停
     断言那两个信源文件不在 D 里

6d 未跟踪碰撞检查再跑一次（与 4.5 同一段代码），命中 → 停

6e A/M/T：git restore --source=<C> --staged --worktree -- <这些路径>
   D：      git rm -f -- <这些路径>
   → 信源文件由 L 一步变成 M；data/** 由 pre_head 内容变成 C 内容

6f 校验：两个信源文件哈希 == M；changed 中其余路径与 C 逐一一致

6g git update-ref refs/heads/master <C> <pre_head>    ← 唯一提交点，compare-and-swap
     （既有写法见 :2638-2646；HEAD 被别人动过直接失败，不会误改）

6h 校验 HEAD == C 且 git diff --cached 为空

6i _restore_owned_stash() 还原脏 data 并 drop

6j 删台账
```

**「合并结果 == 云端现状」的特例**（本机改动其实已经在云端了）：跳过 3~5，只做步骤 6，
返回 `outcome="no_change"` + `merged=True`。重复点两次同步必须命中这条。

#### 载荷说明：未跟踪碰撞预检为什么是 P0

`git merge --ff-only` 遇到会被覆盖的未跟踪文件是**拒绝**；
而 `git restore --worktree` 是**直接覆盖、不警告**。
换了原语之后，同一个碰撞从「同步失败」变成「用户的未跟踪 `data/foo.json` 被静默销毁」。

预检算法：

```
added = git diff --name-only --diff-filter=A -z <pre_head> <fetched_oid>
命中条件：该路径本机未跟踪（git ls-files --error-unmatch 失败）且文件已存在
命中 → 抛 online_sources_untracked_collision（409，details 列路径），零副作用
```

必须在 **push 之前（4.5）与检出之前（6d）各跑一次**。

#### 失败与恢复

`_merge_sync_recovery()` 是**独立函数**：`audit_online_source_operation()` 在 `manual_save`
分支之后紧接着 `if kind == "merge_sync": return _merge_sync_recovery(...)` **提前返回**，
永不进入 `_operation_file_states()` / `_safe_write_rollback_states()` / 通用尾部分支。
`recover_online_source_operation()` 对 merge_sync **显式拒绝** `rollback` 与 `retry_commit`（双保险）。

| 断点 | 台账状态 | 结论 |
|---|---|---|
| A. 步骤 2~3（未建 C） | 无 `operation_commit_oid` | 工作区 JSON 仍为 L → 按 `_manual_save_recovery()` 的「以磁盘 JSON 为准重生成 OPML」修复 → 删台账，用户重新点同步 |
| B. 步骤 4~5（有 C，远端不含 C） | `_remote_contains_commit` 为假 | C 只是游离对象、本机零改动 → 核对工作区仍为 L → 删台账收摊 |
| C. 步骤 5 之后（远端已含 C） | 远端含 C | 唯一动作 `finish_local_fast_forward` = 从 6a 重跑到 6j |

`finish_local_fast_forward` 天然幂等：

- 工作区 == L → 从 6e 做起
- 工作区 == M、HEAD == pre_head → 从 6g 做起
- 工作区 == M、HEAD == C → 从 6i 做起
- 工作区 ∉ {L, M} → 停，文案给出「你的改动已在云端 commit `<C>`，可自行
  `git checkout <C> -- config/online-sources.json` 后重试」

**如实记录的残留窗口**：`git restore` 写文件不保证原子，进程若正好死在写一半，
该文件哈希 ∉ {L, M} → 门禁停在 pending 需人工处理。这与既有 `_restore_owned_stash()`
的风险面同级，方向是 fail-safe（停，不猜）。不额外加机制，只在文案给出可复制的修复命令。

### 4.4 与退订清历史的关系（红线）

同步路径**依旧不调用任何 purge**。合并可能引入「云端已删除某信源」，
此时本机 archive 里该源的历史**保留不删** —— 方向上是 fail-safe。
合并后本机配置改变，下一次「保存」的 `previous_config` 自然是合并后的配置，
不会凭空产生删除对比。必须为这条写死断言测试。
