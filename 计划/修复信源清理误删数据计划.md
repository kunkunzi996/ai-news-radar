## 一、目标（要做成什么样）

修完之后：
1. **GitHub Release 类信源不再被无差别误删**：现在只要保存一次信源配置（哪怕跟 GitHub 毫无关系），所有 GitHub Release 的历史记录都会被判定成"孤儿数据"整体清空。根因是 `scripts/update_news.py` 采集 GitHub Release 时，把 `source` 字段写死成固定文案"GitHub版本订阅"，而不是配置里填的仓库展示名（比如"claude code汉化项目"），导致新写的清理逻辑（按展示名匹配）永远认不出它是"活着的"。修完后，`source` 字段应该用配置里的展示名，这样清理逻辑才能正确识别。
2. **单纯改名字不会被当成删除**：现在清理逻辑只看"保存后的新配置"，不知道"保存前"长什么样。用户只是把某个信源的名字改了一下（没删除任何账号），也会因为"旧名字不在新配置里了"而被当成删除，历史数据被清空。修完后，只要账号本身（配置记录的 `id`，B站动态则是逗号列表里那个人的 UID）在这次保存前后都还在，就算改了展示名，旧名字的历史数据也不会被清掉；但如果账号真的被删掉了（`id`/UID 消失），照样正常清理。

**本次范围只做这两条修复，不做以下事（上次 review 提到的其它问题，本次不动）：**
- 不优化"每次保存信源配置都要扫一遍 5 个数据文件"的性能问题。
- 不改进"清理失败/被跳过时，用户看到的提示和正常成功一模一样"的问题。
- 不修 `purge_tracked_site_ids` 判断 GitHub 信源时"只看 type 字段、不看 id"的比 `update_news.py` 窄的问题（当前配置里没有触发这个坑的记录，属于潜伏风险，不在本次范围）。

### 3.2 `scripts/local_server.py`：改名保护——只在账号真的消失时才清理

**第一步**，在 `alive_source_names_by_site` 函数（约第 723-738 行）**前面**，新增一个"给每个渠道里的账号找一个改名也不变的身份 key"的辅助函数：

```python
def source_identity_names(config: dict[str, Any]) -> dict[str, dict[str, str]]:
    """按 site_id 分组，返回 {site_id: {身份key: 展示名}}。
    wewe_rss / mediacrawler_douyin / mediacrawler_xhs / github_release 是一个账号一条配置记录，
    身份key 用记录的 id；bilibili_dynamic 是多个人共用一条记录（逗号分隔），
    身份key 用逗号列表里每个人自己的 UID（locator）。"""
    identities: dict[str, dict[str, str]] = {site_id: {} for site_id in PURGE_TRACKED_SITE_IDS}
    sources = config.get("sources") if isinstance(config, dict) else None
    if not isinstance(sources, list):
        return identities
    for source in sources:
        if not isinstance(source, dict):
            continue
        site_ids = purge_tracked_site_ids(source)
        if not site_ids:
            continue
        if "bilibili_dynamic" in site_ids:
            names = [part.strip() for part in str(source.get("target") or "").split(",")]
            locators = [part.strip() for part in str(source.get("locator") or "").split(",")]
            for index, locator in enumerate(locators):
                if not locator:
                    continue
                name = names[index] if index < len(names) and names[index] else locator
                identities["bilibili_dynamic"][locator] = name
        for site_id in site_ids:
            if site_id == "bilibili_dynamic":
                continue
            record_id = str(source.get("id") or "").strip()
            display = str(source.get("target") or source.get("name") or "").strip()
            if record_id and display:
                identities[site_id][record_id] = display
    return identities
```

**第二步**，把 `alive_source_names_by_site` 改成同时接收"保存前"的配置，账号身份没变的话把旧名字也算作"活着"：

```python
def alive_source_names_by_site(
    config: dict[str, Any],
    previous_config: dict[str, Any] | None = None,
) -> dict[str, set[str]]:
    """算出每个渠道「现在还认识」的展示名集合：这次保存后的名字都算数；
    另外——同一个账号（id 或 UID 没变）如果只是改了展示名，旧名字这一轮也保留，
    这样单纯改名不会被误判成删除。账号真的被删掉时，旧名字不会被保留。"""
    current = source_identity_names(config)
    previous = source_identity_names(previous_config) if previous_config else {}
    alive: dict[str, set[str]] = {}
    for site_id in PURGE_TRACKED_SITE_IDS:
        names = set(current.get(site_id, {}).values())
        for identity_key, old_name in previous.get(site_id, {}).items():
            if identity_key in current.get(site_id, {}):
                names.add(old_name)
        alive[site_id] = names
    return alive
```

> 安全要点：`previous_config` **默认值必须是 `None`**，且不传时行为要跟改之前完全一致（纯按新配置判断）——这样已有的 `test_alive_source_names_by_site_splits_bilibili_members` 等旧测试不用改也能继续通过。

> 安全要点（务必遵守）：
> - **读旧配置这一步必须放在 `os.replace(tmp_path, self.config_path)` 之前**——一旦执行了 `os.replace`，磁盘上就已经是"新配置"了，再读就读不到"旧配置"了。
