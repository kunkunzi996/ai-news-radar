## 一、目标（要做成什么样）

现状：用户在本地看板「信源配置」页删除一个信源（不管是点整条记录的「删除」按钮，还是在「订阅成员管理」面板里删掉某个 UP 主/公众号/博主/仓库），保存后，`sources.config.json` 里确实不再有这条记录了，但它历史上采集到的文章/视频/动态**依然原样显示**在时间线、Top 榜单里，不会消失。

做完之后的效果：**只要用户点过一次「保存草稿」（信源配置页）或「保存成员」（订阅成员面板），本地的历史数据文件就会立刻把「当前已经不在信源列表里」的那些条目删掉**，不需要等下一次真实联网采集（下一次采集本来也会顺带清理，但慢、而且对「同一渠道下删掉其中一个博主」这种情况其实并不生效，见下面「关键技术点」）。而且这个联动清理是**幂等的**：哪怕用户之前已经删过的信源留下的历史脏数据一直没人清过，只要现在点一次保存（哪怕没改任何东西），也会顺带把这些历史遗留的脏数据一并清掉。

**本次范围只做「保存信源配置时，同步清理本地 `data/*.json` 里属于『已彻底删除信源』的条目」，不做以下事：**
- 不改 `scripts/update_news.py`（采集主脚本）。现有的 `filter_archive_by_source_ids` 是按 `site_id`（渠道级别，比如整条 `wewe_rss` 渠道）过滤的，本来就只在「删掉一整个内置源」时才生效，且要等下次真实采集才生效；这次不动它，只在本地看板保存配置这一步补一层「立即生效」的清理。
- 不处理 `waytoagi` / `hackernews` / `xapi` / `socialdata_x` / `tikhub_*` / `official_ai_sources` / `curated_ai_media_sources` / `opmlrss` 这些"一个 site_id 只对应一整条内置源"的旧内置源被整体删除的情况——这些历史上已经全部删过并且在 `deleted_source_ids` 里了，不是用户现在日常会做的操作；如果以后真的要整个删掉这类源，现有机制（下次点「刷新」）本来就能收尾，不重复造轮子。
- 不动 `data/source-status.json`、`data/merge-log.json`、`data/paid-source-state.json`、`data/title-zh-cache.json`——这些是健康检查/日志/缓存，不是直接展示给用户看的"收集到的信息"，留给下次真实采集自然刷新即可。
- `data/latest-24h.json` 里 `site_stats` 的计数（每个渠道多少条）允许短暂不准，不在本次范围内重新计算——这是统计展示，不是"内容还在不在"的问题。

## 四、后端改动（详细步骤）

### 关键技术点（先看懂这个，再看代码）

`sources.config.json` 里的信源分两种形态：
- **单账号一条记录**：`wewe_rss_maobidao`（猫笔刀公众号）、`mediacrawler_douyin_simon`（抖音博主）、`github_release_javaht_claude_desktop_zh_cn`（GitHub 仓库）等，一条记录 = 一个账号，删记录 = 删账号。
- **多账号共用一条记录**：`bilibili_dynamic_sources` 一条记录里，`target`/`locator` 字段是用英文逗号拼起来的多个人名/UID（如 `"清华姜学长,Koji杨远骋at十字路口,..."`），在「订阅成员管理」面板删掉其中一个 UP 主时，走的是 `assets/app.js` 里的 `setBilibiliSubscriptionMembers()`，只是把逗号列表变短，`bilibili_dynamic_sources` 这条记录本身并不会消失。

而 `data/archive.json` 及各展示文件里每条内容只带两个字段能用来对应信源：`site_id`（渠道类型，如 `"wewe_rss"`、`"bilibili_dynamic"`）和 `source`（展示名，如 `"猫笔刀"`、`"清华学长"`）。`site_id` 是**渠道级别**的，同一渠道下可以有好几个账号共用同一个 `site_id`——这就是为什么现有的 `filter_archive_by_source_ids`（`scripts/update_news.py:3439`，按 `site_id` 整体过滤）对"删掉某渠道下的一个账号"完全不生效：只要这个渠道下还有别的账号活着，`site_id` 就还在"启用"名单里，不会被拦掉。

所以本次要做的匹配规则是：**（site_id, source 展示名）二元组**。对 `wewe_rss` / `bilibili_dynamic` / `mediacrawler_douyin` / `mediacrawler_xhs` / `github_foundation_sunshine_releases`（GitHub Release 订阅的 site_id）这 5 个"同一 site_id 下可能有多个账号"的渠道，从**当前保存的配置**里反查出每个 site_id 下还"活着"的展示名集合（`bilibili_dynamic` 要把 `target` 字段按逗号拆开，因为它是多人共用一条记录）；一条历史数据的 `source` 如果不在对应 site_id 的"活着"集合里，就判定为孤儿数据，直接从文件里删掉。**其余 site_id（比如 `opmlrss`）不在这 5 个之列，本次一律不动**，避免因为它们不是"一账号一记录"的形态而被误杀。

### 4.1 `scripts/local_server.py`：新增匹配与清理函数

放在已有的 `source_config_runtime_ids` 函数（约第 685-702 行）后面、`enabled_source_config_records` 前面。复用 `source_config_runtime_ids` 现成的"根据 id/type/channel/target/locator 文本判断渠道"的写法，不要重复发明一遍判断逻辑。

```python
PURGE_TRACKED_SITE_IDS = frozenset({
    "wewe_rss",
    "bilibili_dynamic",
    "mediacrawler_douyin",
    "mediacrawler_xhs",
    "github_foundation_sunshine_releases",
})


def purge_tracked_site_ids(source: dict[str, Any]) -> set[str]:
    """这条信源记录对应本次要联动清理的哪些 site_id（可能是空集，表示不在追踪范围内）。"""
    ids = set(source_config_runtime_ids(source))
    if str(source.get("type") or "").strip().lower() == "github_release":
        ids.add("github_foundation_sunshine_releases")
    return ids & PURGE_TRACKED_SITE_IDS


def alive_source_names_by_site(config: dict[str, Any]) -> dict[str, set[str]]:
    """给 PURGE_TRACKED_SITE_IDS 里每个渠道，算出当前配置里还存活的展示名集合。
    bilibili_dynamic 记录的 target 是逗号拼接的多个人名，要拆开。"""
    alive: dict[str, set[str]] = {site_id: set() for site_id in PURGE_TRACKED_SITE_IDS}
    sources = config.get("sources")
    if not isinstance(sources, list):
        return alive
    for source in sources:
        if not isinstance(source, dict):
            continue
        site_ids = purge_tracked_site_ids(source)
        if not site_ids:
            continue
        display = str(source.get("target") or source.get("name") or "").strip()
        names = {part.strip() for part in display.split(",") if part.strip()}
        for site_id in site_ids:
            alive[site_id].update(names)
    return alive


def is_item_orphaned(record: dict[str, Any], alive_names: dict[str, set[str]]) -> bool:
    """record 是否属于一个已经彻底删掉的信源。site_id 不在追踪范围内的一律判定为不孤儿（不动）。"""
    site_id = str(record.get("site_id") or "").strip()
    if site_id not in alive_names:
        return False
    source_name = str(record.get("source") or "").strip()
    return source_name not in alive_names[site_id]


def purge_orphaned_from_flat_list(
    items: list[Any],
    alive_names: dict[str, set[str]],
) -> tuple[list[Any], int]:
    kept = [it for it in items if not (isinstance(it, dict) and is_item_orphaned(it, alive_names))]
    return kept, len(items) - len(kept)


def purge_orphaned_from_story_list(
    stories: list[Any],
    alive_names: dict[str, set[str]],
) -> tuple[list[Any], int]:
    """stories-merged.json / daily-brief.json 里的「故事」可能合并了多条来源。
    只要故事里任何一条底层数据属于已删信源，就整条故事一起丢弃（不做部分裁剪重新计分，
    当前数据里 100% 都是单来源故事，做精细裁剪是过度设计；下次真实采集会用剩下的信源重新合并计分）。"""
    kept = []
    removed = 0
    for story in stories:
        if not isinstance(story, dict):
            kept.append(story)
            continue
        members = story.get("items")
        if not isinstance(members, list):
            members = story.get("sources") if isinstance(story.get("sources"), list) else []
        if any(isinstance(member, dict) and is_item_orphaned(member, alive_names) for member in members):
            removed += 1
            continue
        kept.append(story)
    return kept, removed


def write_json_atomic(path: Path, payload: Any, *, compact: bool) -> None:
    text = (
        json.dumps(payload, ensure_ascii=False, separators=(",", ":"))
        if compact
        else json.dumps(payload, ensure_ascii=False, indent=2)
    )
    tmp_path = path.with_suffix(".json.tmp")
    tmp_path.write_text(text, encoding="utf-8")
    os.replace(tmp_path, path)


def purge_deleted_source_data(root_dir: Path, config: dict[str, Any]) -> dict[str, int]:
    """信源配置保存成功后调用：把 5 个本地数据文件里，属于「当前配置里已经不存在」的
    信源账号的历史条目删掉。只处理 PURGE_TRACKED_SITE_IDS 覆盖的渠道。"""
    alive_names = alive_source_names_by_site(config)
    data_dir = root_dir / "data"
    summary: dict[str, int] = {}

    def rewrite_flat(filename: str, list_keys: tuple[str, ...], *, compact: bool) -> None:
        path = data_dir / filename
        if not path.exists():
            return
        try:
            payload = json.loads(path.read_text(encoding="utf-8"))
        except Exception:
            return
        if not isinstance(payload, dict):
            return
        removed_total = 0
        for key in list_keys:
            items = payload.get(key)
            if not isinstance(items, list):
                continue
            kept, removed = purge_orphaned_from_flat_list(items, alive_names)
            payload[key] = kept
            removed_total += removed
        if "total_items" in payload and "items" in list_keys:
            payload["total_items"] = len(payload.get("items") or [])
        if removed_total:
            write_json_atomic(path, payload, compact=compact)
        summary[filename] = removed_total

    def rewrite_stories(filename: str, list_key: str, total_key: str, *, compact: bool) -> None:
        path = data_dir / filename
        if not path.exists():
            return
        try:
            payload = json.loads(path.read_text(encoding="utf-8"))
        except Exception:
            return
        if not isinstance(payload, dict) or not isinstance(payload.get(list_key), list):
            return
        kept, removed = purge_orphaned_from_story_list(payload[list_key], alive_names)
        payload[list_key] = kept
        if total_key in payload:
            payload[total_key] = len(kept)
        if removed:
            write_json_atomic(path, payload, compact=compact)
        summary[filename] = removed

    rewrite_flat("archive.json", ("items",), compact=True)
    rewrite_flat(
        "latest-24h.json",
        ("items", "items_ai", "creator_items_ai", "creator_items_all"),
        compact=False,
    )
    rewrite_flat(
        "latest-24h-all.json",
        ("items_all", "items_all_raw", "creator_items_all"),
        compact=True,
    )
    rewrite_stories("stories-merged.json", "stories", "total_stories", compact=True)
    rewrite_stories("daily-brief.json", "items", "total_items", compact=False)
    return summary
```

> 安全要点（必须遵守）：
> - `write_json_atomic` 必须沿用项目里保存 `sources.config.json` 时同样的「先写 `.tmp`，再 `os.replace`」写法（`do_POST` 里已经这么做了），不要直接用 `write_text` 覆盖原文件，防止写到一半被打断留下半个坏文件。
> - `compact` 参数要跟 `scripts/update_news.py` 里各文件本来的格式保持一致（`archive.json` / `latest-24h-all.json` / `stories-merged.json` 是紧凑格式 `separators=(",", ":")`，`latest-24h.json` / `daily-brief.json` 是 `indent=2`）——这样下次真实采集重新写这些文件时 diff 干净，不会因为格式不一致而整份文件看起来"全变了"。
> - 每个文件只有「确实删了东西」（`removed_total > 0`）才重写，避免每次保存配置都无意义地触碰所有数据文件（也顺带避免不必要的格式化 diff）。
> - 任何一个文件读取失败（不存在、JSON 解析失败、形状不对）都要**跳过而不是抛异常**——这一步是锦上添花的清理，不能因为某个文件损坏就让整个信源保存请求失败。

### 4.2 `scripts/local_server.py`：在保存信源配置成功后调用

修改 `do_POST` 方法里 `/api/source-config` 分支（约第 1776-1813 行）。现有逻辑是：校验 → 写 `sources.config.json` → 返回成功。改成：写文件成功后，**在拿到 `REFRESH_LOCK` 的前提下**才做清理，拿不到就跳过（说明这时候「刷新/一键采集」正在跑，它自己也会重写这几个数据文件，两边同时写会冲突）。

```python
        try:
            raw = self.rfile.read(length)
            payload = validate_source_config(json.loads(raw.decode("utf-8")))
            payload["updated_at"] = payload.get("updated_at") or ""
            body = json.dumps(payload, ensure_ascii=False, indent=2) + "\n"
            tmp_path = self.config_path.with_suffix(".json.tmp")
            tmp_path.write_text(body, encoding="utf-8")
            os.replace(tmp_path, self.config_path)
        except Exception as exc:
            json_response(self, HTTPStatus.BAD_REQUEST, {"ok": False, "error": str(exc)})
            return

        purged_items: dict[str, Any]
        if REFRESH_LOCK.acquire(blocking=False):
            try:
                purged_items = purge_deleted_source_data(self.root_dir, payload)
            except Exception as exc:
                purged_items = {"error": str(exc)}
            finally:
                REFRESH_LOCK.release()
        else:
            purged_items = {"skipped": "refresh_in_progress"}

        json_response(
            self,
            HTTPStatus.OK,
            {
                "ok": True,
                "path": CONFIG_FILENAME,
                "source_count": len(payload.get("sources") or []),
                "purged_items": purged_items,
            },
        )
```

> 安全要点（必须遵守）：
> - 原来写 `sources.config.json` 那个 `try/except` 块**保持不变**，不要把 `purge_deleted_source_data` 塞进同一个 try 里——信源配置本身写成功了就应该算保存成功，清理历史数据失败不该让这次保存报错给用户。
> - 一定要用 `REFRESH_LOCK.acquire(blocking=False)`，不是 `blocking=True`——绝对不能让保存信源配置这个请求卡住等「一键采集」跑完（那个最多能跑 `REFRESH_TIMEOUT_SECONDS`=600 秒），拿不到锁就跳过，下次保存时自然会补上。

2. **验证"删单个账号立即生效"**：打开「订阅成员管理」，选一个当前有多个成员的渠道（比如 B 站动态，或微信公众号），删掉其中一个成员，点「保存成员」。刷新页面（或点"读取结果"），确认这个人/这个号之前发布的内容从时间线、Top 信号里都消失了，同渠道其它人的内容还在。
3. **验证"删整条记录立即生效"**：在「信源配置」页选一条单账号记录（比如某个 GitHub Release 订阅），点「删除」→「保存草稿」，确认这个仓库之前的 Release 记录从看板消失。
4. **验证"没有误删"**：上面两步操作后，抽查 2-3 个**没有删除**的信源，确认它们的历史内容都还完整显示，数量看起来没有异常减少。
5. **验证"不会跟采集打架"**：点一次「一键采集」/「刷新数据」，趁它还在跑的时候，去信源配置页删一个信源并保存，确认这次保存请求本身正常返回成功（不报错、不卡死），可以晚点再保存一次确认清理最终生效即可。

> ⚠️ 第 5 条涉及两个后台动作的时序竞争，单测里没法稳定复现，**必须亲手点一遍**确认没有报错、没有把数据文件写坏（可以在操作前后用 `git status` / `git diff --stat data/` 看一下改动是否合理，而不是文件损坏成空文件或非 JSON）。

## 八、红线（务必遵守）

- 不要新建分支、不要 commit / push（用户验收后自己决定）。
- 不要改本说明清单之外的文件，尤其**不要动 `scripts/update_news.py`**——这次的清理逻辑完全在本地看板这一层（`scripts/local_server.py`）解决，不需要也不应该碰采集主脚本。
