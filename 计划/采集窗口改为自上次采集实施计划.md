## 背景（为什么做这个）

看板的「采集范围」下拉现在有两档：**「过去24小时」**（默认）和**「全量」**。用户每次都是手工点「一键采集」或「刷新看板数据」来采集的，采集间隔不固定（可能隔几小时、也可能隔几天）。固定「过去24小时」这个窗口对手工采集并不准确：

- 如果用户超过 24 小时没采集，两次采集之间那段（24 小时以前、上次采集以后）新发布的内容，就会被「过去24小时」窗口挡在外面、永远进不了看板。
- 如果用户一天内采集了好几次，每次又都按「过去24小时」重新过一遍，做的是重复无用功。

用户的期望：把「过去24小时」这一档，改成**「从上一次采集结束的时间点」开始**——即「这次采集，只接收『距上次采集以来』新发布的内容」。

**现状（技术）**：这个「过去24小时」窗口的唯一来源，是后端 `scripts/local_server.py` 里 `refresh_command()` 在采集范围为 `24h` 时，给采集脚本 `update_news.py` 追加的一个固定参数 `--collect-window-hours 24`。`update_news.py` 收到后，用 `now - 24小时` 作为「本轮新入库项」的发布时间下限来过滤（`filter_raw_items_by_collect_window`，`scripts/update_news.py` 约第 3453 行）。所以本次改造的核心，就是把这个「写死的 24」换成「距上次采集的小时数」。

> 项目根目录：`E:\AI-news-reader\ai-news-radar-run`。下文所有路径均相对该根目录。

**已确认的产品决策**（用户已拍板）：
1. **替换，不是新增**：把「过去24小时」这一档整体改成「自上次采集」；不新增第三档、不保留旧的固定 24 小时档。「全量」档保持不变（第一次补历史时用）。
2. 一键采集和手动「刷新看板数据」**共用同一套逻辑**，两者都跟着变（它们本来就共用 `24h` 这一档）。

## 关键技术点（先看懂这几点，再看代码）

1. **「上次采集结束时间」从哪来——不用新建状态文件。** 每次采集跑完，`update_news.py` 都会把本轮的时间戳写进 `data/source-status.json` 和 `data/latest-24h.json` 的 `generated_at` 字段（当前值例如 `2026-07-05T09:06:36.561958Z`）。而后端处理 `/api/refresh` 请求时，是**先读磁盘、后跑采集**——也就是说，在本轮采集真正覆盖 `source-status.json` 之前，磁盘上的 `generated_at` 还是**上一次**采集的结束时间。直接读它即可，不需要额外存一个「上次采集时间」文件。`scripts/local_server.py` 已有现成的 `read_source_status(root_dir)`（约第 532 行）可以读这个文件。

2. **窗口用「小时数」表示，向上取整，宁大勿小。** 现有参数 `--collect-window-hours` 收的是整数小时。把「距上次采集的时长」换算成小时时**向上取整**（ceil）：比如距上次 3 小时 20 分，就传 4 小时。多算一点点只会让极少量「上次采集时其实已经入过库」的旧条目被重新评估一遍（后续本来就有去重，不会重复展示），**但绝不会漏掉**新发布的内容。反过来如果向下取整就可能漏，所以必须向上取整。

3. **首次采集/读不到时间时，回退到 24 小时。** 如果 `source-status.json` 不存在、或 `generated_at` 缺失/解析失败、或时间戳竟然在「未来」（时钟异常/被手动改过），一律**回退成 24 小时**——等于保留改造前的老行为，安全兜底，绝不报错、绝不让采集本身失败。

4. **本次只改 `refresh_command` 这一处窗口来源。** 抖音/小红书的 MediaCrawler 启动（`start_mediacrawler_platform`，约第 1398 行）虽然也会传 `--collect-window-hours`，但那个参数在 MediaCrawler 侧**只用于跑完后打一行统计日志**（`scripts/run_mediacrawler_douyin.py` 约第 321-331 行，`found N/M rows within Xh`），**并不决定实际抓取多少**（实际抓取量由 `--max-notes` 控制，24h 档下是每个博主最近 5 条）。真正决定「哪些内容被接收进看板」的，全在 `refresh_command` 追加给 `update_news.py` 的那个 `--collect-window-hours`。所以本次**只改 `refresh_command`**，MediaCrawler 启动那处保持原样（详见「本次不做」第 1 条）。

### 4.2 `scripts/local_server.py`：新增三个函数

定位 `normalize_collection_scope`（约第 1609 行）和 `refresh_command`（约第 1618 行）之间，在 `refresh_command` 的 `def` **前面**插入下面三个函数：

```python
def last_collection_time(root_dir: Path) -> datetime | None:
    """读上一次采集结束时间：来自 data/source-status.json 的 generated_at。
    文件不存在 / 字段缺失 / 解析失败都返回 None，交给调用方兜底。"""
    status = read_source_status(root_dir)
    if not isinstance(status, dict):
        return None
    raw = str(status.get("generated_at") or "").strip()
    if not raw:
        return None
    try:
        parsed = datetime.fromisoformat(raw.replace("Z", "+00:00"))
    except ValueError:
        return None
    if parsed.tzinfo is None:
        parsed = parsed.replace(tzinfo=timezone.utc)
    return parsed


def resolve_collect_window_hours(
    scope: str,
    last_collected_at: datetime | None,
    now: datetime,
    *,
    default_hours: int = 24,
) -> int:
    """算出这轮采集应接收「最近多少小时内新发布」的内容。
    scope=all -> 0（不设窗口，保留全部抓到的内容）。
    scope=24h -> 距上次采集的小时数，向上取整；上次时间缺失/无法解析/不在过去时，
    回退成 default_hours（默认 24），等同改造前的老行为。"""
    if scope == COLLECTION_SCOPE_ALL:
        return 0
    if last_collected_at is None:
        return default_hours
    seconds = (now - last_collected_at).total_seconds()
    if seconds <= 0:
        return default_hours
    return max(1, math.ceil(seconds / 3600))


def collect_window_hours_for_scope(
    root_dir: Path,
    scope: str,
    *,
    now: datetime | None = None,
) -> int:
    now = now or datetime.now(timezone.utc)
    return resolve_collect_window_hours(scope, last_collection_time(root_dir), now)
```

> 复用提示：
> - `read_source_status(root_dir)` 是文件里已有的函数（约第 532 行），直接复用，别自己重新拼 `data/source-status.json` 路径。
> - `datetime` / `timezone` 已经在文件顶部 `from datetime import datetime, timezone`（约第 19 行）导入，直接用。
> - `COLLECTION_SCOPE_ALL`（值 `"all"`）是已有常量（约第 33 行），直接用。
>
> 安全要点（务必遵守）：
> - `last_collection_time` 里读文件 / 解析时间的每一步都要能**静默返回 `None`**（`read_source_status` 读不到、`generated_at` 为空、`fromisoformat` 抛 `ValueError`），绝不能让「读上次时间」这一步的异常影响到采集本身。
> - `resolve_collect_window_hours` 里 `seconds <= 0`（上次时间等于或晚于现在，属时钟异常）必须走回退分支返回 `default_hours`，不能返回 0（返回 0 会被下游当成「全量、不设窗口」，语义完全错位）。
> - **把「纯计算」和「读磁盘」拆成两个函数**（`resolve_collect_window_hours` 纯计算、`collect_window_hours_for_scope` 读盘）是刻意的，为的是纯计算部分能不碰磁盘、不依赖真实时间地做单测（见第六节）。请保持这个拆分，别合并成一个读盘函数。

    root_dir: Path,
    collection_scope: str = COLLECTION_SCOPE_24H,
    *,
    now: datetime | None = None,
) -> list[str]:
    scope = normalize_collection_scope(collection_scope)
