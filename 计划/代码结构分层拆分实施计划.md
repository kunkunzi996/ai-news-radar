## 背景（为什么做这个）

项目三个热点文件已经膨胀到难以维护的程度（均已实测行数）：

| 文件 | 行数 | 问题 |
|---|---|---|
| `scripts/update_news.py` | 8151 | 约 25 个信源抓取器 + 工具函数 + 归档/合并/打分管线 + 800 行 `main()` 全塞一个文件 |
| `assets/app.js` | 5994 | 308 个顶层函数共享一个文件：渲染、订阅、已阅、源配置、本地采集面板全耦合 |
| `scripts/local_server.py` | 2513 | 2100 行模块级辅助函数 + HTTP Handler + main 混在一起 |

**现状（已读真代码确认的关键事实，执行时不得违背）：**

1. **双入口模式**：`update_news.py` 既被测试以 `from scripts.update_news import ...` 导入（`scripts/__init__.py` 已存在，是包），又被 `local_server.py:1750` 以文件路径 `python scripts/update_news.py --args` 直接运行。文件头部已有 `try: from scripts.ai_relevance import ... except ModuleNotFoundError: from ai_relevance import ...` 的双导入兼容。
2. **测试安全网**：`tests/` 下 14 个文件、209 个用例（`test_local_server` 占 64 个），几乎全部 `from scripts.update_news import ...` / `from scripts.local_server import ...` 按名导入——只要门面 re-export 齐全，测试导入语句一行都不用改。
3. **仅有 3 处 mock 补丁点需要跟着搬**（已 grep 全量确认，没有第 4 处）：
   - `tests/test_private_bridge_sources.py:75` → `patch("scripts.update_news.requests.get", ...)`
   - `tests/test_topic_filter.py:265` → `patch("scripts.update_news.HN_ALGOLIA_QUERIES", ...)` 和 `patch("scripts.update_news.time.sleep")`
   - `tests/test_local_server.py` → `patch("scripts.local_server.active_bilibili_cdp_port", ...)`
4. **前端无构建工具**：`index.html:338` 用 `<script src="./assets/app.js?v=..." defer>` 加载经典脚本（非 module）。经典脚本按序执行、共享全局作用域，因此**按序拆成多个经典脚本文件是零语义变更**——唯一约束是"加载时执行的顶层语句"顺序。`app.js` 顶层结构：1–207 行为 `state` 对象与 DOM 常量声明，**第 208 行有唯一一处顶层跨区调用 `state.readItemIds = loadReadItemIds()`**，中段全是函数声明，5798–5994 行为事件绑定和 `init()`。
5. 前端缓存靠 query 版本号（当前 `?v=local-server-restart-0707a`），改动 JS 必须换新版本号。

**已确认的决策：**
1. 纯结构搬移，**不改任何函数体、不改任何行为**；发现疑似 bug 只记录汇报，不顺手修。
2. 后端拆成 `scripts/radar/` 一个总包；`update_news.py` 和 `local_server.py` 保留为门面 + 入口，对外路径（测试导入、子进程调用、启动命令）全部不变。
3. 前端**不引入 ES Modules、不引入构建工具**，拆成按序加载的经典脚本，全局作用域照旧。
4. 每阶段一个验收闸门，用户提交后再进下一阶段；任何时刻测试不绿不得继续搬。

## 二、关键技术点（需要动的硬骨头）

1. **入口路径统一** → `update_news.py` 直接运行时 `sys.path[0]` 是 `scripts\`，导不进 `scripts.radar`。门面文件开头加统一垫片（替代逐模块 try/except）：

```python
import sys
from pathlib import Path

if __package__ in (None, ""):  # python scripts/update_news.py 直接运行
    sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
```

之后门面和包内模块一律用 `scripts.radar.xxx` / `scripts.ai_relevance` 绝对导入，不再需要双导入 try/except。`local_server.py` 门面加同一段垫片。

2. **mock 补丁点迁移** → 上面列的 3 处补丁改成函数搬入后的新模块路径（例如 `scripts.update_news.HN_ALGOLIA_QUERIES` → `scripts.radar.fetchers.public.HN_ALGOLIA_QUERIES`）。原理：`patch` 打的是"函数实际读取全局名的那个模块"，门面里的副本打了没用。每改一处立刻单跑对应测试文件确认。
3. **门面 re-export** → 门面用 `from scripts.radar.common import *  # noqa: F401,F403` 逐模块导入。包内每个模块**不需要**定义 `__all__`（wildcard 默认导出全部非下划线名，含常量），但要注意先导 `common` 再导依赖它的模块，避免同名覆盖（搬移时不改名，理论上无同名）。
4. **前端顶层副作用** → `app.js:208` 的 `state.readItemIds = loadReadItemIds();` 在拆分后会因 `loadReadItemIds` 尚未加载而报错，必须移进 `boot.js` 的 `init()` 调用之前（这是全文件唯一一处需要挪位置的语句，其余照原文搬）。
